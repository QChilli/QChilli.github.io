动态窗口

理解这个动态窗口可以从它的设计目的和运行机制两个角度来分析。动态窗口是为了确保系统只处理最近一段时间（比如3秒）内的数据，并通过一个环形缓冲区来实现数据的添加和移除。而且动态窗口可以看作是FIFO（先进先出）或者队列的一种实现形式，尤其是在处理实时数据或流数据时，动态窗口的工作方式和队列或FIFO的行为非常相似。

动态窗口的机制是为了确保数据的实时性和有效性，通过环形缓冲区和读写指针的管理，保证系统在有限的内存中处理最近的（例如3秒）数据，并能及时丢弃过期数据。这种设计广泛用于需要处理连续数据流的场景，例如传感器数据采集、网络流量监控等。

动态窗口的设计目的

  1. 实时性数据处理：系统需要在一个固定的时间范围内（如3秒）处理数据，而不是无限制地存储所有历史数据。动态窗口保证了系统只对最近一段时间的数据进行计算或分析，保证数据的实时性。
  2. 节省内存：通过环形缓冲区的设计，系统不需要无限制地扩展内存，只需要一定大小的缓冲区来存储最近的数据。超出时间窗口的数据会被移除，腾出空间给新数据。
  3. 滑动窗口：每当有新数据进入时，窗口会“滑动”——添加新数据的同时，移除超出时间范围的数据，保证窗口内的数据始终是最新的。

动态窗口的运行机制

  1. 环形缓冲区：
    • 环形缓冲区的设计允许在有限的空间内进行数据的“读”和“写”操作。
    • buffer_start_r 是读指针，表示当前窗口的起始位置，指向最旧的数据。
    • buffer_end_w 是写指针，表示当前窗口的结束位置，指向最近写入的数据。
  2. 添加新数据：
    • 当新数据被采集后，它会被放入到 buffer_end_w 位置，并更新写指针 buffer_end_w，将其移到下一个可写入的位置。
    • 如果写指针追上了读指针（即缓冲区满了），系统就需要调整 buffer_start_r 来丢弃最旧的数据，从而保证新数据有空间写入。
  3. 移除超时数据：
    • 每次写入新数据后，系统会检查缓冲区中的最旧数据（buffer_start_r）是否已经超出了设定的时间窗口（例如3秒）。
    • 通过一个 while 循环，移除所有时间戳超过3秒的旧数据，直到缓冲区内的数据全部在最近3秒内为止。

本质

  1. 动态窗口本质上是一个滑动窗口：当新数据写入时，窗口会自动“滑动”，旧数据会被移除，新数据会被加入。窗口的大小（例如3秒）是固定的，但窗口内的数据内容会随时间变化。
  2. 读写指针的管理：环形缓冲区通过读写指针来控制数据的读写，确保不会出现数据覆盖或错误。buffer_start_r（读指针）负责指出当前有效数据的最早位置，buffer_end_w（写指针）则指向下一个写入位置。
  3. 时间范围的维护：通过对比当前时间与数据时间戳，可以有效地维护3秒这个时间窗口，确保在每次添加新数据时，只保留最近3秒内的数据。

FIFO与动态窗口异同

动态窗口与FIFO/队列的相似点

  • 先进先出：动态窗口中的数据处理顺序是按时间顺序进行的，最早进入窗口的数据在超出时间范围或队列满时最先被移除，这与FIFO和队列的先进先出特性一致。

  • 顺序性：无论是动态窗口还是FIFO/队列，都保证了数据按照进入的顺序进行处理，新数据会按顺序添加，旧数据按顺序移除。

动态窗口与FIFO/队列的不同点

  • 时间限制：队列和FIFO通常不考虑时间因素，而动态窗口的核心是按时间来管理数据。数据是否被移除不仅取决于它是否是最早的，还取决于它是否超出了时间窗口的范围。

  • 内存管理：FIFO或队列可以是动态大小，而动态窗口通常使用固定大小的环形缓冲区，限制存储的数据量。如果缓冲区满了，新数据将覆盖最旧的数据。

主要改动与分析

在你的原始代码中，似乎有两个主要问题，导致了 (*nd_change_arr)++ 进入一个不断递增的循环：

  1. 状态标志变量未区分 A 和 B 通道：

  2. 明确的条件和标志变量设置：
    现在你为 A 和 B 通道分别处理了两个逻辑条件，确保了 freqMax - freqMin < param_arr[0] 的情况被独立地判断。这样，如果当前频率波动符合设定条件，标志位会被设置为 1，并且在 (*nd_change_arr)++ 执行后状态重置为 0。通过这种状态机式的设计，标志位必须先通过 “突变发生” 再通过 “突变消失” 的过程，这样逻辑变得更加清晰和严谨。

  3. 不同统计数组的分开处理：
    你还将 nh_change_arr 和 nd_change_arr 的处理逻辑分开得更加清楚，并且使用了 statistical_array[2] 和 statistical_array[3] 来区别不同的变化处理。这样确保了各个变化情况（如 A 通道和 B 通道的频率突变）是独立判断的，不会相互影响。由于数组引用和标志的单独管理，逻辑不再重叠，从而避免了多次累加的问题。

为什么现在只会加一次？

现在的逻辑可以理解为一个 有限状态机，它有两个状态：

  • 状态 0：没有发生突变。
  • 状态 1：检测到突变并准备计数。

在进入 check_freq_data() 函数时：

  • 如果符合条件（freqMax - freqMin < param_arr[0]），状态从 0 变为 1，表示检测到了频率突变。
  • 在下一次进入这个函数时，如果之前状态是 1 并且频率已经不再突变（else if (Pd_A_mutation_status == 1)），则 (*nd_change_arr)++ 执行，并将状态重置为 0。

因此，每个通道在频率突变时只会有一次状态变化，并且在突变消失后才会执行累加操作。这个状态重置的设计确保了不会在每次调用时反复累加。

总结

你之所以成功修复了这个问题，主要是因为：

  1. 你对 A 通道和 B 通道使用了独立的状态标志（Pd_A_mutation_status 和 Pd_B_mutation_status）。
  2. 状态标志的明确区分，使得每个突变状态只会触发一次 (*nd_change_arr)++，而不是反复累加。
  3. 通过这种状态控制的方式，有效避免了逻辑混乱导致的条件反复触发。

这样清晰的状态管理可以确保系统按预期工作，避免任何的累加错误。继续保持良好的编程实践，状态机的管理是非常重要的概念，特别是在实时信号处理这样的场景中。

  1. 在函数调用间保持值不变：将 temp_old 声明为 static 可以确保它的值在多次调用 led_disp() 时保持不变。如果不加 static，每次调用该函数时 temp_old 都会重新初始化为 0xff。由于 temp_old 用来存储上一次的 LED 状态，这种值的保持是非常重要的。只有当 temp（新的 LED 状态）不同于 temp_old 时，才会更新 temp_old，这使得 led_disp() 可以有效地检测到 LED 状态的变化。
  2. 限制变量的作用域：虽然 temp_old 在函数调用间保留其值，但由于 static 的作用域规则，它仍然只在 led_disp() 函数内部有效。这可以防止代码其他部分修改 temp_old，从而减少潜在的错误或冲突。

总结来说，这里的 static 是为了在多次函数调用中保留上一次的 LED 状态，同时将变量的作用域限制在 led_disp() 函数内。

sizeof

sizeof()里放数组名和struct+结构体名是不是就能准确的算出数组和结构体的大小？

sizeof() 运算符在 C 和 C++ 中确实可以用于计算数组和结构体的大小：

  1. 数组：当 sizeof() 中放入数组名（例如 sizeof(array)）时，它会返回整个数组的总大小（单位是字节）。例如，对于 int array[10];，sizeof(array) 会返回 10 * sizeof(int) 的值，这表示整个数组所占的内存空间。如果传递的是指针而不是数组名，比如 int *array_ptr，那么 sizeof(array_ptr) 得到的只是指针本身的大小（通常是 4 或 8 字节），而不是数组的总大小。

  2. 结构体：sizeof(struct StructName) 或 sizeof(structure_variable) 可以准确地计算结构体所占的内存大小。sizeof 会考虑结构体内部字段的对齐（alignment），并且会将所有字段的大小加在一起，因此返回的值就是结构体占用的总字节数。

所以，将数组名和结构体名传给 sizeof() 确实能准确计算出数组和结构体的内存大小。这种方法在编译期直接计算，效率高且准确。sizeof()确实能准确计算其字节大小，包括'\0'等所有元素。

  • __FILE__ 是预定义宏，它表示当前编译单元的文件路径，具体值由编译器决定。
  • 在大多数编译器中：
    • 绝对路径: __FILE__ 包含完整的文件路径。例如：C:/project/src/main.c。
    • 相对路径: 如果文件是通过相对路径传递给编译器，__FILE__ 的值可能是相对路径，例如：src/main.c。

如果 __FILE__ 中的路径分隔符是正斜杠 /，例如：/home/user/main.c，那么：

总线异常（Bus Fault）

1.外部中断

按键未按下导致外部中断触发的问题与清不清中断标志位没关系，但仍然会进入总线访问异常中断。总线访问异常（Bus Fault）通常与访问非法地址、栈溢出或其他内存错误有关。

1.原因

  1. 外部中断配置不当：

    • 如果外部中断配置（例如 EXTI0）或中断向量表的处理不正确，可能会触发不可预期的行为，导致访问非法内存区域。特别是在外部中断服务程序（ISR）中进行非法操作时。
    • 如果没有正确配置中断的优先级或中断处理函数中访问了非法内存（如未初始化的指针、NULL 指针、未对齐访问等），会引发总线访问异常。

  2. 外设的中断触发条件：

    • 即使按键没有按下，中断仍然可能被触发，尤其是如果外部中断线的状态没有正确检查或配置。
    • 比如，如果 GPIO 输入引脚的电平状态未正确处理，可能会导致外部中断触发。若该中断服务函数（ISR）未正确清除中断标志位或处理中断时访问了无效地址，可能会导致异常。

  3. 中断服务函数访问非法内存：

    • 在你的 EXTI0_IRQHandler 函数中，如果在处理中访问了无效的内存地址或使用了未初始化的指针，程序会进入总线访问异常。

  4. 外设寄存器配置问题：

    • 如果访问了未初始化或无效的外设寄存器，可能会导致总线访问异常。例如，在你的代码中，如果中断源没有正确配置或清除，外设可能未正确初始化，导致访问外设时发生异常。
