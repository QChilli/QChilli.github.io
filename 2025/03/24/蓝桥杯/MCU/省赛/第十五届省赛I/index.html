1.数码管闪烁

1.错误示范

本次试题并不涉及数码管闪烁，只可能是底层和逻辑功能代码的问题。

程序进入定时器中断服务函数，Freq_Cal 一开始系统默认是 0 ，此时程序执行 else 分支，这时才能检测到 Freq_Cal 为负数，而由于程序设计实物的时候，原本应该 CheckOut_Error 立刻为 1 ，现在必须先为 0 等到下一次进入中断服务函数后，才能使 CheckOut_Error 为 1 ，还要 Freq_Cal 重新被赋值为 0 ，所以才会使 CheckOut_Error 一直 0 / 1 交替，使得数码管会闪。

2.解决方法

1.先计算 Freq_Cal 再逻辑判断（错误）

不可行：

  • Freq_Cal 最后计算的结果为负数，也没有必要保留这个负数结果，直接清 0 就好。
  • Freq_Cal 计算代码执行后，万一立刻改变校验值 Para_CheckOut_Proc ，且迅速切换到频率界面时，频率值会延时更新，导致数据更新实时性降低，影响系统性能。

2.先判断 Para_CheckOut_Proc 的值是否为负数（正确）

2.C语言数据类型的隐式转换

1.错误示范

根据 C 的整数提升规则和算术转换规则可知：

  1. 整数提升：如果操作数是小于 int 的类型（如 short），会被提升为 int。在这个例子中，Freq_Cal 已经是 int，所以不需要提升。
  2. 类型统一：如果两个操作数的类型不同，编译器会将其中一个类型转换为另一个类型。通常：
    • 如果一个操作数是无符号整数 (unsigned int)，另一个是有符号整数 (int)，则有符号整数会被转换为无符号整数。
    • 转换后，两个操作数都变成 unsigned int，然后进行比较。

当 Freq_Cal 是负数时，Freq_Cal > Freq_Max 中Freq_Cal（int 类型）会被隐式转换为 unsigned int，编译器直接将其二进制位模式重新解释为无符号整数值，也就是说，负数会被“环绕”到无符号整数的上限范围。
